CAFlow — Document & Practice Management

A production-grade SaaS for Indian chartered accountants. Replaces the WhatsApp chaos of document collection with an organized, client-facing portal — plus practice management features that sit beyond what Tally covers.

3
User portals
18
Core modules
P1
MVP: 8 modules
~14w
Solo dev estimate
What this is not: This is not an accounting tool. CAs will keep using Tally for bookkeeping and ITR software for filing. We own the workflow that happens before filing — client communication, document collection, case tracking, fee management. That is our moat.

Build phases

Phase 1 — MVP (weeks 1–10)
Auth & onboarding, Client management, Case system, Document engine, Client portal (token-based upload + download). This is enough to charge ₹800/month and solve the core problem.
Phase 2 — Retention (weeks 11–20)
Deadline & compliance calendar, Fee & billing module, Client communication log, Team management. These make CAs feel the product is essential — not just useful.
Phase 3 — Moat (weeks 21–30)
WhatsApp notifications (outbound only), Tally export, Digital signature requests, Admin portal, Multi-firm support. This is where you defend against copycats.
Solo founder rule: Do not start Phase 2 until you have 20 paying CAs using Phase 1 daily. Revenue validates direction. Building features nobody asked for is how solo SaaS projects die.
Technical foundation

System architecture

Every layer, every service, every technology decision — with the reasoning behind it.

🧱
Frontend — React + Vite
Three apps, one codebase, shared component library
Three separate Vite apps in a monorepo: apps/ca-portal, apps/client-portal, apps/admin. Shared UI components in packages/ui. Shared types in packages/types.
  • React 18 + Vite 5 — fast HMR, ESM builds, tree-shaking
  • React Router v6 with lazy loading per route — code splitting out of the box
  • TanStack Query v5 — server state, caching, optimistic updates, background refetch
  • Zustand — lightweight client state (auth, UI state, active workspace)
  • React Hook Form + Zod — form validation with TypeScript-first schema
  • TailwindCSS v3 — utility classes, design tokens in tailwind.config
  • Shadcn/ui — copy-paste components built on Radix — accessible, unstyled base
  • Tanstack Table v8 — headless data tables with sort, filter, pagination
  • React Dropzone — drag-and-drop file upload with preview
  • Date-fns — date manipulation, deadline calculations, IST timezone handling
apps/
  ca-portal/ — main CA dashboard, team inbox, case management
  client-portal/ — tokenised upload/download, no login required
  admin/ — internal account management (Phase 2)
packages/
  ui/ — shared Shadcn components, design tokens
  types/ — shared TypeScript interfaces for API contracts
⚙️
Backend — Node.js + Hono
REST API, webhook processing, background jobs
  • Hono.js — ultra-fast Node.js framework, TypeScript-first, edge-ready if needed later
  • Postgres via Supabase — managed DB, Row Level Security for multi-tenancy, realtime subscriptions
  • Drizzle ORM — type-safe SQL queries, migrations, schema definition in TypeScript
  • BullMQ + Redis (Upstash) — background jobs: send notifications, process file uploads, schedule reminders
  • Cloudflare R2 — S3-compatible file storage, zero egress cost, signed URLs for secure access
  • Resend — transactional email (document received, case complete, deadline reminder)
  • Razorpay — subscriptions, payment links, GST invoices (India-first)
  • Clerk — auth for CA portal (email OTP, Google SSO, team management)
  • Zod — request/response validation on every API route
🗄️
Multi-tenancy model
How data is isolated between CA firms
Every CA firm is a workspace. Every table has a workspace_id column. Supabase Row Level Security policies ensure that a query from Workspace A can never return data from Workspace B — even if a bug exists in application code.
  • JWT contains workspace_id + user_id + role
  • Every API route extracts workspace_id from JWT — never from query params
  • RLS policy on every Postgres table: workspace_id = auth.jwt()->>'workspace_id'
  • File storage: every file path prefixed with workspaces/{workspace_id}/
  • Client portal tokens encode workspace_id + case_id — no login, but fully scoped

Client Portal — Module 1

Tokenised access system

The client never creates an account. They receive a link, tap it, and they are in their own secure space. No friction, works for all age groups.

This is the design decision that makes the product work for Tier 2 and Tier 3 India. No login = no abandonment at the auth screen.
🔑
How tokens work
Security without friction
  • Token generated on server: crypto.randomBytes(32).toString('hex')
  • Stored in DB: token → workspace_id + client_id + case_id + expiry
  • URL: caflow.app/c/[token] — this IS the authentication
  • Token expires: 14 days for document requests, 30 days for download links, never for client vault access (if client logs in with OTP)
  • Token is single-use for sensitive operations (e.g. onboarding form), multi-use for document upload
  • If token expired: show "This link has expired. Please contact your CA." with CA's phone number
  • For returning clients who prefer login: optional phone OTP login to see all their cases across time
